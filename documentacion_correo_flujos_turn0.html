TURN0 - Documentacion para exponer

Correo de bienvenida, registro y flujos por usuario

Esta guia explica lo trabajado hoy: el envio automatico de correo al registrarse, el problema con Hotmail/prueba, como se corrigio, y como intervienen los roles administrador, vendedor/negocio y cliente dentro del sistema.

1. Resumen simple

Que se queria lograr

Se queria que cuando una persona se registre desde la pagina visual de TURN0, el sistema tome el correo escrito en el formulario y envie automaticamente un mensaje de bienvenida a ese mismo correo.

Estado final: el registro desde la interfaz crea el usuario y manda el correo de bienvenida al correo que la persona escribio en el formulario.
Importante: antes existia una variable llamada WELCOME_TEST_EMAIL. Esa variable hacia que todos los correos de prueba llegaran a un solo correo fijo, por ejemplo Hotmail. Eso servia para pruebas, pero no era lo que se queria para el registro real. Por eso se cambio para que el registro use directamente user.email.
2. Flujo del correo

Que pasa cuando un usuario presiona Registrarse

Este es el recorrido completo desde la pagina hasta el correo:

El usuario entra a la pagina de registro.
Escribe nombre, correo, contrasena y tipo de usuario.
Presiona el boton Registrarse.
React captura el evento del formulario.
React llama a la API con Axios.
Express recibe la ruta /api/auth/register.
El backend crea o recupera el usuario.
El backend llama al servicio de correo.
Nodemailer envia el mensaje al correo escrito por el usuario.
La pagina muestra si el correo fue enviado o si hubo advertencia.
3. Frontend

Donde esta el formulario de registro

El formulario visual esta en:

frontend/src/pages/RegisterPage.jsx

Funcion principal del registro

Esta funcion se ejecuta cuando el usuario presiona el boton Registrarse. 

async function handleSubmit(event) {
  event.preventDefault();
  setMessage("");

  if (!form.fullName || !form.email || !form.password || !form.confirmPassword || !form.role) {
    setMessageType("text-danger");
    setMessage("Completa todos los campos.");
    return;
  }

  const response = await registerUser({
    fullName: form.fullName,
    email: form.email,
    password: form.password,
    role: form.role
  });
}

Explicacion para exponer

Esta funcion primero evita que el navegador recargue la pagina. Luego valida que los campos no esten vacios. Si todo esta bien, manda los datos al backend usando la funcion registerUser.

Donde se llama a la API desde el frontend

frontend/src/api/authService.js

export function registerUser(payload) {
  return apiClient.post("/auth/register", payload);
}

Aqui se envia la informacion del formulario hacia el backend. La ruta completa termina siendo:

http://localhost:4000/api/auth/register

Donde esta configurado Axios

frontend/src/api/apiClient.js

export const apiClient = axios.create({
  baseURL: import.meta.env.VITE_API_URL || "http://localhost:4000/api",
  headers: {
    "Content-Type": "application/json"
  }
});

Axios es el puente entre React y Express. React no habla directamente con MySQL; primero llama al backend por medio de Axios.

4. Backend

Donde llega la peticion de registro

La ruta del backend esta en:

backend/src/routes/auth.routes.js

router.post("/register", register);

Esto significa que cuando el frontend llama a /api/auth/register, Express ejecuta la funcion register.

Controlador del registro

backend/src/controllers/auth.controller.js

export async function register(req, res, next) {
  try {
    const user = await registerUser(req.body);
    let welcomeEmail = { sent: false, reason: "NOT_ATTEMPTED" };

    try {
      welcomeEmail = await sendWelcomeEmail(user);
    } catch (mailError) {
      welcomeEmail = { sent: false, reason: "MAIL_SEND_FAILED" };
    }

    res.status(201).json({
      message: user.alreadyRegistered
        ? "El usuario ya existia. Se reenvio el correo de bienvenida."
        : "Usuario registrado correctamente",
      user,
      welcomeEmail
    });
  } catch (error) {
    next(error);
  }
}

Explicacion para exponer

El controlador primero registra el usuario. Luego intenta enviar el correo. Si el correo falla, el sistema no rompe el registro; solo devuelve una advertencia. Esto es importante porque no queremos que una falla de correo impida crear la cuenta.

5. Registro o reenvio

Que pasa si el usuario ya existe

Se pidio que si el usuario ya estaba registrado, igual se pudiera enviar el correo otra vez. Esa logica esta en:

backend/src/services/auth.service.js

const [existingUsers] = await pool.query(
  "SELECT id, full_name, email, role FROM users WHERE email = ?",
  [normalizedEmail]
);

if (existingUsers.length > 0) {
  const existingUser = existingUsers[0];
  return {
    id: existingUser.id,
    fullName: existingUser.full_name,
    email: existingUser.email,
    role: existingUser.role,
    alreadyRegistered: true
  };
}

Antes, si el correo ya existia, el backend devolvia error. Ahora no crea duplicado, sino que recupera el usuario existente y permite reenviar el correo de bienvenida.

6. Servicio de correo

Donde se envia el correo

El servicio de correo esta en:

backend/src/services/mail.service.js

Linea mas importante

const recipient = user.email;

Esta linea es clave. Significa que el correo de bienvenida se envia al correo que el usuario escribio en la interfaz, no a un correo fijo del archivo .env.

Configuracion de Nodemailer

const transporter = nodemailer.createTransport({
  host: env.mail.host,
  port: env.mail.port,
  secure: env.mail.port === 465,
  auth: {
    user: env.mail.user,
    pass: env.mail.pass
  }
});

Nodemailer usa los datos SMTP para conectarse al correo emisor. En este caso, el emisor configurado puede ser Gmail.

Envio real del correo

await transporter.sendMail({
  from,
  to: recipient,
  subject,
  html
});

Aqui se manda el correo real. El campo to recibe el correo del usuario registrado.

7. Plantilla HTML del correo

Como esta estilizado el mensaje de bienvenida

La plantilla tambien esta en:

backend/src/services/mail.service.js

function buildWelcomeTemplate(user) {
  const roleName = user.role === "dueno_negocio"
    ? "vendedor / negocio"
    : "cliente";

  return `
    <h1>Bienvenido, ${escapeHtml(user.fullName)}</h1>
    <p>Tu cuenta como ${roleName} fue creada correctamente.</p>
    <h2>Nuestros planes para TURN0</h2>
  `;
}

Esta plantilla tiene diseno con colores, titulo, mensaje de bienvenida, planes de la empresa y un boton para entrar a TURN0.

Contenido del mensaje: agradece al usuario, explica que puede hacer en TURN0, menciona los planes de la empresa y ofrece un boton para entrar al sistema.
8. Variables de correo

Donde se configura el correo emisor

Las credenciales se colocan en:

backend/.env

SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USER=tu_correo@gmail.com
SMTP_PASS=clave_de_aplicacion
SMTP_FROM="TURN0 <tu_correo@gmail.com>"
WELCOME_TEST_EMAIL=
Importante: para Gmail no se usa la contrasena normal. Se usa una clave de aplicacion generada desde la cuenta de Google.

En el registro real, WELCOME_TEST_EMAIL ya no controla el destinatario. El destinatario es el correo escrito en el formulario.

9. Prueba manual vs registro real

Diferencia entre test:mail y registro desde la pagina

Accion Como funciona Para que sirve

10. Flujos por rol

Como interviene cada tipo de usuario

Administrador

Supervisa clientes Supervisa citas Supervisa turnos Revisa reportes Descarga PDF

El administrador tiene una vista completa. Puede revisar la operacion general, validar el dashboard, mirar reportes y controlar la informacion principal del sistema.

Vendedor / negocio

Registra clientes Agenda citas Gestiona turnos Consulta reportes

El vendedor o dueno de negocio usa TURN0 como herramienta diaria. Su flujo principal es crear clientes, organizar citas, atender turnos y revisar como va el negocio.

Cliente

Se registra Recibe correo Crea cita Ve cronometro

El cliente entra al sistema para gestionar sus citas. Despues de registrarse recibe el correo de bienvenida. Luego puede consultar sus citas y ver cuanto falta para su proxima atencion.

11. Como exponerlo

Guion breve para explicar

Puedes decirlo de esta manera:

En TURN0, cuando un usuario se registra desde la interfaz, React captura los datos del formulario y los envia al backend usando Axios. El backend recibe la ruta de registro, crea o recupera el usuario en MySQL y luego llama a un servicio de correo. Ese servicio usa Nodemailer para enviar una plantilla HTML de bienvenida al correo que el usuario escribio en la pagina. Si el usuario ya existia, no se duplica; se reutiliza su informacion y se reenvia el correo de bienvenida.
